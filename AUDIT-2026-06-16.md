### M1. `decision_only` parameter is stored but never enforced — behavioral contract is a lie

**File:** `src/index.ts` L85-93 (DispatchParams), L298-301 (stored to stateJson), `src/event-classifier.ts` L26-35  
**Severity:** MUST FIX — documented tool behavior does not match implementation

**What's wrong:**  
`langgraph_dispatch` accepts `decision_only: boolean` (default `true`). The tool docstring says:

> "When true (default), only decision/milestone/terminal events wake the agent; status events update flow state silently."

The implied corollary is that `decision_only=false` changes wake behavior (presumably suppressing milestone wakes). But `decision_only` is stored in `stateJson` and passed as metadata to LangGraph — it is *never read back* by `processEvent` or `classifyEvent`. The classifier always maps `milestone → wake-light → wake` regardless of this flag.

**Concrete failure mode:**  
An agent or workflow author who reads the docstring and sets `decision_only=false` to suppress milestone noise gets zero behavioral change. They believe they've reduced wake frequency; they have not. Every node step still fires a full agent turn. The flag is a documentation promise with zero enforcement.

More subtly: `decision_only=true` is described as the default, and the current behavior *matches* that description. So `decision_only=true` is silently correct. Only `decision_only=false` is broken. Anyone who *doesn't* set the flag doesn't notice; anyone who tries to change it does.

**What to do:**  
Option A (preferred): Define what `decision_only=false` means concretely — e.g., "milestone events do NOT wake; only decision/terminal/hitl do" — implement it by passing the flag through to `processEvent` (via `WebhookHandlerDeps` or by reading it from flow `stateJson`), add it to `classifyEvent` or add a wrapper check before calling `wakeAgent`, and test both branches.

Option B: Remove the parameter from `DispatchParams` and the docstring entirely until the behavior is defined and implemented. An unused parameter that changes nothing is worse than no parameter.

### M2. Orphaned "queued" flow when dispatch fails — creates permanent stale state

**File:** `src/index.ts` L324 (`flows.createManaged`), L390-410 (runIdPromise rejection + catch)

**What's wrong:**  
In `langgraph_dispatch`, `flows.createManaged()` is called *before* `runIdPromise` resolves. If the LangGraph server is unreachable, the network call in `dispatchAndStream` fails immediately, `onError` fires, `runIdPromise` rejects, and the catch block at L397 returns an error result to the agent. The flow created at L324 is **never cleaned up** — it remains in `"queued"` status indefinitely.

**Concrete failure mode:**  
After a failed dispatch (LangGraph server down, wrong URL, network partition):

1. `langgraph_inspect()` returns this zombie "queued" flow as the "latest flow"
2. `langgraph_resume()` finds `candidate.status !== "waiting"` → returns `flow_not_waiting`
3. Re-dispatching creates a second "queued" zombie on top of the first
4. `flows.findLatest()` always returns the newest zombie, hiding the older ones

This is the same class of stale-state bug as BINGO-8. The plugin creates observable side-effects (flow records) and doesn't clean them up on failure. After three failed dispatches the session's flow list is polluted with orphaned "queued" records.

**What to do:**  
In the `catch` block (L397-410), tombstone the orphaned flow before returning the error:

```typescript
try {
  flows.finish({
    flowId: flow.flowId,
    expectedRevision: flow.revision,
    stateJson: {
      terminal_title: "dispatch_failed",
      terminal_summary: message,
    },
    endedAt: Date.now(),
  });
} catch {
  // best-effort cleanup; don't shadow the original error
}
```

Add a test: dispatch with a fetch that immediately rejects → `langgraph_inspect` returns a finished/failed flow, not a queued one.

### M3. `LanggraphClient.resumeRun()` sends wrong LangGraph wire format — silent wrong behavior if called

**File:** `src/langgraph-client.ts` L89-115

**What's wrong:**  
`resumeRun()` calls `createRun()` with:
```typescript
input: { command: { resume: resumePayload } }
```
This sends `{"input": {"command": {"resume": ...}}}` to LangGraph's `POST /threads/{tid}/runs`.

The correct LangGraph wire format for a Command resume is:
```json
{"command": {"resume": ...}}
```
with `command` as a **top-level** key, not nested under `input`. Compare `dispatchAndStream` in `event-subscriber.ts` (L357-367), which correctly sends `command` at the top level.

The method is dead code now — `langgraph_resume` routes through `dispatchAndStream` since Phase 5 — but it sits in a public-facing client class. It is wrong in a way that is very hard to debug: LangGraph accepts the POST and starts a *new run* with `{"command": {"resume": ...}}` as the initial input state. The workflow doesn't get resumed; it gets a new invocation with unexpected input. No error is returned.

**What to do:**  
Delete `resumeRun()` since it's unused. If it needs to be preserved for future non-streaming resume use, fix the wire format:

```typescript
async resumeRun(threadId, assistantId, resumePayload, opts?) {
  // correct: command is top-level, NOT inside input
  const body: Record<string, unknown> = {
    assistant_id: assistantId,
    command: { resume: resumePayload },
  };
  if (opts?.metadata) body.metadata = opts.metadata;
  const res = await this.fetch(`/threads/${encodeURIComponent(threadId)}/runs`, {
    method: "POST", body,
  });
  // ...
}
```

Add a test asserting the request body does NOT contain `input` when called.

### M4. Concurrent `langgraph_resume` calls can open duplicate SSE streams on the same thread

**File:** `src/index.ts` L464-492 (waiting-flow lookup), L539-590 (dispatchAndStream call)

**What's wrong:**  
There is no locking around the "find waiting flow → open SSE subscriber" sequence in `langgraph_resume`. Two concurrent calls (e.g., the agent fires two resume tool calls in the same turn, or the same session receives two parallel HITL replies) will both:
1. Read `candidate.status === "waiting"` (both pass the guard)
2. Call `dispatchAndStream` independently
3. Each opens `POST /threads/{tid}/runs/stream` — creating **two separate LangGraph runs** on the same thread

Both SSE subscribers then call `processEvent` independently for their respective event streams. `wakeAgent` fires from each subscriber's terminal/hitl events, producing duplicate agent turns on the same session.

**Why this matters:**  
In a threaded Slack channel, a user could trigger this by typing a reply and clicking "retry" before the first resume completes. In a Slack DM, the agent itself could retry a failed resume call. This causes two parallel LangGraph runs, each processing the same thread's state, with unpredictable results.

**What to do:**  
Add a `Set<string>` at plugin scope to track in-progress resumes by flow ID:

```typescript
const resumeInProgress = new Set<string>();
// in langgraph_resume execute, after checking status === "waiting":
if (resumeInProgress.has(candidate.flowId!)) {
  return jsonResult({
    status: "error",
    reason: "resume_already_in_progress",
    message: `Flow ${candidate.flowId} already has a resume in flight.`,
  });
}
resumeInProgress.add(candidate.flowId!);
// clear in finally block (wrap runIdPromise + flows.resume in try/finally)
```

### M5. SSE terminal + webhook terminal both call `flows.finish()` — webhook 500 causes LangGraph retry storm

**File:** `src/index.ts` L364-380 (onClose synthetic terminal), `src/webhook-handler.ts` L173-194 (terminal case in processEvent), `src/index.ts` L430-434 (webhook route registration)

**What's wrong:**  
When `callbackPublicBaseUrl` is configured, the plugin passes the webhook URL to LangGraph at dispatch time (L303-305). LangGraph then delivers the terminal event via **both** the SSE stream AND the webhook POST. Both paths independently call `processEvent`, which calls `flows.finish()`.

- First path (SSE): `flows.finish()` succeeds, `wakeAgent()` fires.  
- Second path (webhook): `flows.finish()` throws a revision conflict (flow is already finished), exception propagates up through `buildHandler`'s try/catch, which returns HTTP **500** to LangGraph.

LangGraph interprets a 500 as a transient error and **retries** the webhook. Each retry hits the same 500 cycle. Whether LangGraph caps retries or backs off forever depends on its version-specific retry policy — which the plugin has no control over.

The problem is symmetric: if the webhook delivers the terminal first, the SSE path's `processEvent` call throws on `flows.finish()`, gets caught by the `onEvent` wrapper (L348-357), logs a warning, and continues. The wake was already fired from the webhook path — no duplicate wake, but a confusing warning log that looks like data loss.

**What to do:**  
Make `processEvent` idempotent for duplicate delivery. In the `terminal` case, check the current flow status before calling `finish()`:

```typescript
case "terminal": {
  const current = flows.get(body.flow_id);
  const alreadyDone = ["succeeded", "failed", "completed"].includes(
    String((current as { status?: string })?.status ?? "")
  );
  if (alreadyDone) {
    // idempotent: already finished, ack without error
    break;
  }
  flows.finish({ ... });
  break;
}
```

The webhook handler should return 200 (not 500) when the flow is already in a terminal state. Similarly guard `setWaiting` for duplicate `hitl` delivery.

### S2. `AbortController` from `dispatchAndStream` is silently discarded — no way to cancel in-flight streams

**File:** `src/index.ts` L341 (dispatch), L544 (resume); `src/event-subscriber.ts` L290 (dispatchAndStream return type)

**What's wrong:**  
`dispatchAndStream(...)` returns an `AbortController` for cancelling the in-flight SSE stream. In both `langgraph_dispatch` and `langgraph_resume`, the return value is discarded:

```typescript
dispatchAndStream({ ..., handlers: { ... } });  // AbortController dropped
```

There is no mechanism to cancel an in-flight stream. If the gateway is restarting, if the flow is cancelled externally, or if the tool call receives an AbortSignal, the SSE stream runs until LangGraph closes it — up to the LangGraph server's own run timeout. Multiple parallel long-running streams accumulate without any lifecycle management.

**What to do:**  
Store the AbortController in a plugin-scoped `Map<string, AbortController>` keyed by flow ID. When a flow transitions to terminal state (in processEvent or in an explicit cancel path), look up and call `.abort()`. Clear from the map in `onClose`. This also gives `langgraph_inspect` something useful to report ("stream active: yes/no").

### S3. `decision` events leave no trace in flow state — silent audit gap

**File:** `src/webhook-handler.ts` L143-147

**What's wrong:**  
The `case "decision":` branch is empty — no flow mutation before `wakeAgent` fires:

```typescript
case "decision":

**Why it's a problem:**  
The flow's `currentStep` and event history have no record that a decision event was received. case "decision":

    task: title,
    status: "completed",
    startedAt: Date.now(),
    lastEventAt: Date.now(),
    progressSummary: summary || null,
  });
  break;
```

### S4. PII logged at `info` level: Slack user IDs in session keys

**File:** `src/wake-agent.ts` L134-136

**What's wrong:**  
`logger.info` emits:
```
langgraph-bridge: wakeAgent dispatched agent=main sessionKey=agent:main:slack:direct:u03rvtn37e2 msglen=312
```

Session keys encode Slack user IDs (e.g., `u03rvtn37e2`). These appear in `gateway.log` in plaintext at info level, persisted to disk. In a multi-user deployment, the gateway log becomes a registry of which Slack users triggered LangGraph workflows.

**What to do:**  
Redact the session key in log output. Log only the structural prefix:

```typescript
const logSk = params.sessionKey.split(":").slice(0, 4).join(":") + ":[redacted]";
logger?.info?.(`langgraph-bridge: wakeAgent dispatched agent=${params.agentId} sessionKey=${logSk} msglen=${params.message.length}`);
```

The full session key is already present in flow state for debugging; it doesn't need to be in the high-frequency wake log.

### S5. `wakeAgent` spawns one subprocess per event with no backpressure — amplification risk

**File:** `src/wake-agent.ts`, `src/webhook-handler.ts` L200-215

**What's wrong:**  
Every call to `wakeAgent` synchronously spawns a new `openclaw agent` subprocess via `execFile`. Since `processEvent` is called once per SSE frame (which arrives once per workflow node), a workflow with 8 nodes emits up to 8 milestone events → 8 concurrent subprocess invocations, each:
- Opening a WebSocket to the gateway
- Initiating an LLM call
- Waiting up to 600s for the agent turn to complete
- Posting to Slack

With the default `DEFAULT_TURN_TIMEOUT_MS = 600_000`, those 8 subprocesses each hold their socket for up to 10 minutes. There is no concurrency cap, no coalescing, no "is there already a wake in progress for this session" guard.

**Concrete impact:**  
A fleet workflow with designer, coder, reviewer, qa, merger, summarizer (6 nodes × 2 events each = 12 milestones) generates 12 agent turns. 12 Slack posts from a single `langgraph_dispatch`. This is both expensive and noisy for the user.

**What to do:**  
Add a per-session in-flight wake semaphore. Track active wake subprocess count per sessionKey. If a wake is already in progress, either skip the new one (for milestone) or queue it (for terminal/decision/hitl). As an immediate improvement, `wakeAgent` could check and drop duplicate wakes within a 1-second window for the same session+kind.

### S6. Out-of-order event delivery can corrupt flow state — `seq` is accepted but never checked

**File:** `src/webhook-handler.ts` L78 (`seq?` on IncomingEventBody), DESIGN.md Phase 4

**What's wrong:**  
The `seq` field is present on `IncomingEventBody` but is never validated against the flow's last-seen sequence number. If a network delay causes a `terminal` (seq=5) event to arrive before a `hitl` (seq=3) event:
1. `flows.finish()` fires, flow moves to terminal
2. `flows.setWaiting()` fires, flow moves back to waiting

Whether step 2 succeeds depends on whether the SDK allows `setWaiting` on a finished flow. If it does, the agent is woken to answer a HITL gate that LangGraph already completed. If it doesn't, an exception is thrown, the wake is suppressed, and the HITL is silently dropped.

DESIGN.md acknowledges this as a Phase 4 item, but "we'll fix it later" is not documentation for the fact that it's actively dangerous when used with a high-latency webhook path.

**What to do:**  
Track `last_seq` in flow `stateJson`. In `processEvent`, before any mutation:

```typescript
const lastSeq = Number((stateJson?.last_seq as number | undefined) ?? -1);
if (body.seq !== undefined && body.seq <= lastSeq) {
  deps.logger?.warn?.(`langgraph-bridge: dropping out-of-order event flow=${body.flow_id} seq=${body.seq} last_seen=${lastSeq}`);
  return { status: "ok", action: "skipped:out-of-order" };
}
```

Update `last_seq` on every processEvent call.

### S7. No auth on outbound LangGraph calls — blocks LangGraph Cloud / any production deployment

**File:** `src/langgraph-client.ts` L135-145 (private `fetch`), `src/event-subscriber.ts` L330-344 (dispatchAndStream headers)

**What's wrong:**  
All HTTP calls to the LangGraph server are unauthenticated. `LanggraphClient.fetch()` sets only `content-type: application/json` with no `Authorization` header. `dispatchAndStream` similarly only sets `content-type` and `accept`. The plugin `ConfigSchema` has no `langgraphApiKey` field.

**Why it's a problem:**  
LangGraph Cloud requires `X-Api-Key: <key>` on all requests. Any multi-tenant LangGraph deployment behind an API gateway will require auth. If this plugin is ever used outside the private EC2 VLAN, all calls will fail with 401/403. The failure mode is not gracefully surfaced — `dispatchAndStream` treats the non-OK response as a generic stream error.

**What to do:**  
Add `langgraphApiKey?: string` to `ConfigSchema` and `PluginConfig`. Thread it through `LanggraphClient` constructor and add it as a header in both `LanggraphClient.fetch()` and `dispatchAndStream`:

```typescript
if (opts.apiKey) {
  headers["x-api-key"] = opts.apiKey;
}
```

### S8. Gateway restart mid-flow: flows stuck in "running"/"queued" forever — no on-startup reconciliation

**File:** `src/index.ts` register() function; DESIGN.md Phase 4

**What's wrong:**  
When the gateway restarts, all in-flight SSE subscribers are killed (the IIFE in dispatchAndStream terminates with the process). Any flow in `"running"` or `"queued"` status remains there indefinitely. On restart, `register()` runs but opens no SSE subscribers for existing flows.

BINGO-8 was an instance of exactly this problem: the plugin showed a flow as "running" while LangGraph had completed or was blocked at a HITL. Recovery required manually inspecting LangGraph state and calling `langgraph_inspect` / manual flow manipulation.

DESIGN.md lists "Daily reconciliation cron" as Phase 4, but there is no on-startup reconciliation either.

**What to do:**  
In `register()`, after setting up the webhook route, add a reconciliation step:

```typescript
// On startup: scan for flows in running/queued state and attempt SSE reattach
const allFlows = api.runtime.tasks.managedFlows.list?.();  // check SDK availability
for (const flow of allFlows ?? []) {
  if (flow.status !== "running" && flow.status !== "queued") continue;
  const state = parseMaybeJson(flow.stateJson);
  const threadId = state?.langgraph_thread_id as string | undefined;
  if (!threadId) continue;
  // ... reopen SSE subscriber for this thread
}
```

At minimum, add a startup log line that counts flows in non-terminal states so an operator knows to investigate: `"langgraph-bridge: startup found N flows in running/queued state — no SSE reattach (manual reconciliation required)"`.

### S9. Milestone wake fires on every node step — not documented, no way to disable

**File:** `src/event-classifier.ts` L27 (`milestone → wake-light`), `src/webhook-handler.ts` L196-212 (`actionRequiresWake`)

That's 12 milestone wakes for a complete run. Combined with the no-backpressure issue (S5), this means 12 parallel agent turns, 12 Slack posts.

The `decision_only` parameter was supposed to address this (see M1), but since it's not enforced, there is currently no way to configure "only wake me for decisions and terminal." **What to do:**  
Implement `decision_only` properly (ties to M1). Until then, at minimum document in the README and tool description that every milestone event fires a separate agent turn, and warn workflow authors to be conservative with milestone emission rate.

### G2. Log lines don't carry a consistent set of correlation IDs

**File:** `src/webhook-handler.ts` L215, `src/wake-agent.ts` L134, `src/index.ts` L412, L425

Some log lines include `flow=`, `thread=`, `run=`. Others include only one. None consistently include all four of `flow_id`, `thread_id`, `run_id`, `session_key` (redacted).

Define a `formatLogCtx` helper and use it everywhere:
```typescript
function formatLogCtx(ctx: { flowId?: string; threadId?: string; runId?: string }) {
  return [
    ctx.flowId   ? `flow=${ctx.flowId}`   : null,
    ctx.threadId ? `thread=${ctx.threadId}` : null,
    ctx.runId    ? `run=${ctx.runId}`     : null,
  ].filter(Boolean).join(" ");
}
// Usage: `langgraph-bridge: routed ${formatLogCtx({flowId, threadId})} kind=${kind}`
```

This lets `grep flow=f-1234` pull the complete trace across all log lines for a flow.

### G3. `dispatchAndStream` should use a discriminated union for `input` vs `command`

**File:** `src/event-subscriber.ts` L235-258 (`StreamingDispatchParams`)

Currently `input` and `command` are both optional. A caller can pass both (the comment says "command takes precedence — input is ignored"), which is confusing. TypeScript can enforce mutual exclusivity:

```typescript
type StreamingDispatchParams = {
  baseUrl: string; threadId: string; flowId: string; assistantId: string;
  metadata?: Record<string, unknown>;
  handlers: StreamHandlers;
  fetchImpl?: typeof fetch;
} & (
  | { input: Record<string, unknown> | null; command?: never }
  | { command: { resume?: unknown; [k: string]: unknown }; input?: never }
);
```

This makes the `command` vs `input` choice explicit at compile time and removes the need for the runtime `command ? ... : ...` branch comment.

## ✅ WHAT THE PLUGIN DOES WELL

These are genuine strengths, not participation trophies.

---

### W1. `processEvent` is pure and unit-tested — HTTP wiring is a thin shell

The webhook handler separates concerns cleanly: `buildHandler` owns auth, body limit, and HTTP plumbing; `processEvent` owns all routing logic and takes injected deps. The 238-line test file unit-tests `processEvent` for all five event kinds, reply-hint formatting, summary truncation, and agentId plumbing — without any HTTP scaffolding. This design made it easy to test the Phase 4 wake pivot in isolation.

---

### W2. Atomic create+stream eliminates the TOCTOU race window

Using `POST /threads/{tid}/runs/stream` (atomic create+stream) instead of `createRun` + `GET /runs/{rid}/stream` eliminates the race where a fast run completes before the subscriber connects. The comment in `event-subscriber.ts` L1-50 explains exactly why this matters and what was rejected. This is the kind of non-obvious architectural choice that is invisible if it works and catastrophic if missed.

---

### W3. Webhook doesn't trust the inbound session key — looks it up server-side

In `buildHandler`, the session key is never read from the POST body. It's looked up from the flow record via `lookupFlow(deps, body.flow_id)`. This means even if an attacker has a valid `callbackToken` and a valid `flow_id`, they cannot redirect a wake to a different session by spoofing the session key in the body. The routing is sealed server-side.

---

### W4. Token-absent-means-refuse is the right defensive default

The webhook handler treats a missing `callbackToken` config as a hard refusal (returns 503) rather than accepting all requests. A misconfigured plugin produces a loud, obvious failure rather than silent open ingress. This is the correct posture for a production webhook endpoint.

---

### W5. `wakeAgent` is inject-testable, never throws, and fire-and-forget by design

`execFile` is injected through `WakeAgentDeps`, tested with a mock, and explicitly documented as fire-and-forget with no-throw semantics. The three early-return guards (no agentId, no sessionKey, no message) and the subprocess-failure test case are all present. The decision to always return 200 to LangGraph even if the wake fails is the correct tradeoff: LangGraph should not retry webhooks because OpenClaw's internal CLI hiccupped.

---

### W6. `normalizeResumePayload` is pure, exported, and thoroughly tested

The HITL normalization is a pure function with 11 test cases covering all keyword aliases, the `<decision>: <feedback>` split form, non-keyword pass-through, object pass-through, and the empty/whitespace edge cases. Exporting it from `index.ts` means it's testable without the full plugin SDK.

---

### W7. `parseSseFrame` handles CRLF and LF explicitly, with regression tests

The LangGraph dev server emits CRLF-terminated SSE lines. The tests explicitly cover both `\n` and `\r\n` separator styles. This is the kind of cross-platform detail that causes silent parse failure in production if missed.

---

### W8. Bearer-token comparison happens before any body parsing

The webhook handler authenticates before attempting to read the body. This means unauthenticated requests are rejected without consuming the CPU of JSON parsing. The order (method check → auth check → body read → schema check → flow lookup) is correct.

## 🧪 OBSERVATIONS DURING AUDIT

These are things noticed but not easily classified as MUST FIX or SHOULD FIX.

---

### O1. `actionHeartbeatReason()` in `event-classifier.ts` is dead code after the Phase 4 pivot

**File:** `src/event-classifier.ts` L56-59

`actionHeartbeatReason()` returns the `reason` string that was passed to `requestHeartbeat(...)` before the Phase 4 pivot to `wakeAgent`. It's exported, never called by any non-test code, and references the old wake primitive. Not harmful; just dead export that will confuse anyone reading the file.

---

### O2. `DESIGN.md` phase plan is significantly out of sync with shipped reality

The doc lists Phase 3 (langgraph_inspect, inbound_claim) as not complete. `langgraph_inspect` shipped in Phase 3; `inbound_claim` was tried and pivoted to the `langgraph_resume` tool in Phase 5. Phase 4 (production hardening — reconciliation, retry, per-run audit log) is listed as future but is shipped code now. A new developer reading DESIGN.md would have a significantly wrong mental model of what's done.

---

### O3. `smoke-webhook.ts` script is listed in `package.json` scripts but the file doesn't exist

**File:** `package.json` L13, `scripts/` directory

`npm run smoke:webhook` references `scripts/smoke-webhook.ts` which does not exist in the scripts directory (`ls scripts/` shows only `smoke-langgraph.ts` and `smoke-streaming.ts`). The script will fail with a file-not-found error if someone runs it.

---

### O4. `__interrupt__` detection relies on a LangGraph implementation detail

**File:** `src/event-subscriber.ts` L109-133

The HITL detection in `classifyStreamFrame` checks for the literal node name `"__interrupt__"`. This is an internal LangGraph convention, not a documented public API. It was correct as of LangGraph 0.10.0, but LangGraph has historically changed internal node names between versions. If this changes, HITL detection silently breaks — `__interrupt__` updates become milestones, the flow never moves to waiting, and the agent is never asked the HITL question. There's no test that catches this beyond a unit test that hard-codes the same assumption.

---

### O5. Wake message length could exceed CLI argument limits for long summaries + thread hints

**File:** `src/webhook-handler.ts` L222-239 (formatEventText)

The wake message is assembled as: `[reply-hint line]\n[head line]\n[summary (up to 280 chars)]`. With a thread reply-hint (~180 chars) + head line (~60 chars) + summary (280 chars), the message can reach ~520 chars. If the `openclaw agent --message` CLI has a message length limit (not documented), this would be silently truncated at the CLI level, potentially cutting off the reply-hint or summary. Worth documenting the known safe range.

---

### O6. `langgraph_inspect` uses `fromToolContext` in dispatch vs `bindSession` in the inspect tool itself

**File:** `src/index.ts` L308 (dispatch), L161 (inspect tool execute)

The dispatch path uses `api.runtime.tasks.managedFlows.fromToolContext({sessionKey, deliveryContext})`. The inspect and resume tools use `api.runtime.tasks.managedFlows.bindSession({sessionKey})`. Whether these two APIs have different semantics for reads/writes is undocumented. If `fromToolContext` scopes writes differently than `bindSession`, flows created via dispatch might not be readable by inspect.

---

### O7. README says "Status: Phase 0 scaffold" — has not been updated since initial commit

**File:** `README.md` L7

The README still says "Phase 0 scaffold. Ships a single stubbed tool." The plugin is at Phase 5+ with three tools, a webhook route, streaming SSE, and resume support. The README's architecture diagram still shows `requestHeartbeat` as the wake primitive (removed in Phase 4). A new developer reading the README would have completely wrong expectations about the plugin's capabilities.

---

### O8. `peerDependencies` has only a lower bound — no upper bound on OpenClaw version

**File:** `package.json` L19: `"openclaw": ">=2026.5.17"`

No upper bound means the plugin loads against any future OpenClaw version. Plugin SDK compatibility is not guaranteed across major versions. Without CI running the test suite against a range of OpenClaw versions, a breaking SDK change will be discovered at upgrade time in production.

---

### O9. Thread reply-hint (v0.11.0) is documented as a workaround — is the upstream issue tracked?

**File:** `src/index.ts` L284-289 (comment on dispatch tool), `src/webhook-handler.ts` L257-276 (buildReplyHint)

The comment on `langgraph_dispatch` says the reply-hint is "a workaround pending OpenClaw runtime support." DESIGN.md does not mention tracking an upstream issue. If this is a gap in the OpenClaw `openclaw agent` CLI (no `--thread-id` flag), it should be filed as an issue against OpenClaw so the workaround can eventually be removed.
